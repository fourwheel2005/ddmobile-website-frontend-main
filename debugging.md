# Debugging Skill — ศาสตร์การไล่บั๊กฉบับ Fable

> Debugging ไม่ใช่การเดาแล้วลองสุ่ม ๆ — มันคือ **วิธีการทางวิทยาศาสตร์**: ตั้งสมมติฐาน → หาหลักฐาน → ตัดตัวเลือก → พิสูจน์

---

## 1. กฎเหล็ก 5 ข้อ

1. **Reproduce ให้ได้ก่อน** — บั๊กที่ reproduce ไม่ได้ = แก้ไม่ได้ (ได้แค่เดา) ทำให้มันพังซ้ำได้ในขั้นตอนที่สั้นที่สุด
2. **อ่าน error message ทั้งหมด จริง ๆ** — ช้า ๆ ทีละบรรทัด รวมถึง stack trace บรรทัดล่าง ๆ คำตอบอยู่ในนั้นบ่อยกว่าที่คิด และให้สนใจ error **ตัวแรก** ไม่ใช่ตัวสุดท้าย (error หลังมักเป็นผลพวง)
3. **เปลี่ยนทีละอย่าง** — แก้ 3 จุดพร้อมกันแล้วหาย จะไม่มีวันรู้ว่าอะไรคือสาเหตุ (และอีก 2 จุดอาจสร้างบั๊กใหม่)
4. **อย่าเชื่ออะไรที่ยังไม่ได้พิสูจน์** — "ส่วนนี้ไม่น่าเกี่ยว" คือประโยคที่พาหลงทางบ่อยสุด บั๊กชอบอยู่ในจุดที่เรามั่นใจว่าถูก
5. **แก้ root cause ไม่ใช่อาการ** — ถ้าแก้ด้วยการเพิ่ม `if null then skip` โดยไม่รู้ว่าทำไมมันเป็น null ได้ = ซุกบั๊กไว้ใต้พรม

## 2. Workflow มาตรฐาน

```
1. REPRODUCE   ทำให้พังซ้ำได้อย่างเสถียร + จดขั้นตอน
2. MINIMIZE    ตัดปัจจัยจนเหลือ case เล็กสุดที่ยังพัง
               (ลบข้อมูลครึ่งหนึ่ง ยังพังไหม? ปิด feature X ยังพังไหม?)
3. LOCATE      หาว่าพังที่ "ชั้นไหน" ก่อน แล้วค่อยหา "บรรทัดไหน"
4. HYPOTHESIZE ตั้งสมมติฐานที่ทดสอบได้: "ถ้า X เป็นสาเหตุ แล้วทำ Y ต้องเห็น Z"
5. TEST        ทดลองที่แยกแยะได้ว่าสมมติฐานถูกหรือผิด
6. FIX         เขียน test ที่ reproduce บั๊ก (แดง) → แก้ → test เขียว
7. REFLECT     บั๊กแบบเดียวกันซ่อนที่อื่นอีกไหม? ทำไม test เดิมไม่จับ?
```

### เทคนิค LOCATE ที่ทรงพลังที่สุด: Binary Search
- **ในโค้ด**: จุด A ข้อมูลยังดี จุด B ข้อมูลพังแล้ว → เช็คตรงกลาง → ตัดครึ่งไปเรื่อย ๆ
- **ใน history**: `git bisect` — บอก commit ที่ดีกับที่พัง เดี๋ยว git หา commit ต้นเหตุให้ใน log₂(n) ขั้น
  ```bash
  git bisect start
  git bisect bad HEAD
  git bisect good v1.2.0
  # หรืออัตโนมัติ: git bisect run pytest tests/test_broken.py
  ```
- **ในข้อมูล**: input 10,000 แถวพัง → ลองครึ่งแรก/ครึ่งหลัง → หดจนเหลือแถวเดียวที่เป็นตัวการ
- **ใน environment**: ปิด extension/plugin/middleware ทีละครึ่ง

## 3. เครื่องมือตามสถานการณ์

### 3.1 Print/Log debugging (ดูถูกไม่ได้ ใช้ให้เป็น)
- Print ให้มี **ป้ายกำกับ + ค่า + type**: `print(f"[order.py:42] total={total!r} type={type(total)}")` — เพราะ `"5"` กับ `5` print เหมือนกัน
- Print **ก่อนและหลัง** จุดต้องสงสัย เพื่อจับว่าค่าเพี้ยนตรงไหน
- ใส่ marker ที่หาง่าย (`>>> DEBUG`) จะได้ลบออกหมดตอนจบ

### 3.2 Debugger (คุ้มค่าการเรียนรู้เสมอ)
- Breakpoint + step ดูค่าจริง ดีกว่า print 20 รอบ: `breakpoint()` (Python), `debugger;` (JS), IDE breakpoint
- **Conditional breakpoint**: หยุดเฉพาะเมื่อ `order_id == "ORD-8812"` — สำคัญมากตอนบั๊กเกิดกับ 1 ใน 10,000 รายการ
- Watch expression ดูค่าที่เปลี่ยนระหว่าง step

### 3.3 ตามประเภทปัญหา
| อาการ | เครื่องมือ/วิธี |
|---|---|
| HTTP/API เพี้ยน | เปิด network tab / `curl -v` ยิงตรง แยกให้ออกว่า client หรือ server ผิด |
| ช้า | Profiler ก่อนเสมอ (py-spy, clinic.js, pprof) — อย่าเดา hotspot |
| Memory รั่ว | Heap snapshot เทียบ 2 จุดเวลา หาว่า object อะไรโตไม่หยุด |
| พังเฉพาะ production | อ่าน log + error tracking (Sentry), เทียบ env/config/data ต่างจาก local ตรงไหน |
| Race condition | เพิ่ม logging พร้อม timestamp/thread id, ลองบีบ thread pool เหลือ 1 แล้วดูว่าหายไหม (ถ้าหาย = ใช่) |
| ทำงานเพี้ยนหลัง deploy | `git diff` ระหว่าง version ที่ดีกับที่พัง, git bisect |
| "ไม่ได้แก้อะไรเลยแต่มันพัง" | มีอะไรเปลี่ยนแน่นอน: dependency อัปเดต? ข้อมูลเปลี่ยน? cert หมดอายุ? disk เต็ม? วันที่/เวลา? |

## 4. บั๊กยอดฮิตและกลิ่นของมัน

| อาการ | สาเหตุที่พบบ่อยสุด |
|---|---|
| พังบางครั้ง ไม่พังบางครั้ง | race condition, ลำดับ async, test แชร์ state, เวลา/timezone, hash order |
| พังเฉพาะข้อมูลบางตัว | edge case ใน data: null, empty, unicode, ค่าซ้ำ, ยาวผิดปกติ |
| พังเฉพาะครั้งแรก / เฉพาะครั้งที่สอง | cache, lazy init, state ค้างจากรอบก่อน, migration ยังไม่รัน |
| พังเฉพาะ production | env var ต่าง, ข้อมูลจริงใหญ่/สกปรกกว่า, ไม่มี debug mode มาช่วยกลบ, proxy/load balancer |
| ตัวเลขเพี้ยนนิด ๆ | floating point, ปัดเศษคนละจุด, timezone, integer division |
| ช้าลงเรื่อย ๆ | memory leak, ตารางโตแล้ว query ไม่มี index, connection ไม่ถูกปิด |
| Off-by-one | ขอบ loop, slice, pagination, inclusive/exclusive |

## 5. เมื่อติดแหง็ก (stuck > 30 นาที)

1. **Rubber duck** — อธิบายปัญหาออกเสียง/พิมพ์ให้ละเอียดเหมือนเล่าให้คนไม่รู้เรื่องฟัง — สมองจะเจอรูตอนเรียบเรียง
2. **ตั้งคำถามกับ assumption ทีละข้อ**: "ไฟล์นี้ถูก load จริงไหม? แน่ใจว่ารัน version ล่าสุด? เช็ค DB ถูกตัวไหม? cache ล้างหรือยัง?" — เขียนรายการแล้วพิสูจน์ทีละข้อ
3. **เช็คของโง่ ๆ ก่อนของฉลาด**: save ไฟล์หรือยัง, restart server หรือยัง, typo ในชื่อตัวแปร/env var, รันอยู่ branch ไหน
4. เดินออกไปพัก — วิธีแก้ชอบมาตอนไม่ได้นั่งจ้อง
5. เขียน minimal reproduction แยกโปรเจกต์ — ครึ่งหนึ่งของกรณีจะเจอคำตอบระหว่างเขียน

## 6. หลังแก้เสร็จ (ขั้นที่คนข้ามแล้วเจ็บ)

```
□ มี test ที่ fail ก่อนแก้และผ่านหลังแก้ (regression test)
□ เข้าใจ "ทำไม" มันพัง ไม่ใช่แค่ "แก้แล้วหาย" — ถ้าอธิบาย root cause ไม่ได้ = ยังไม่จบ
□ grep หา pattern เดียวกันที่อื่นในโค้ดเบส — บั๊กชอบมีฝาแฝด
□ ลบ debug code / print / log ชั่วคราวออกหมด
□ ถ้าบั๊กหลุดถึง production: คิดว่า guard ชั้นไหนควรจับได้ (test? type? lint? monitoring?) แล้วเพิ่มชั้นนั้น
```
