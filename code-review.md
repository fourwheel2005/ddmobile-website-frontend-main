# Code Review Skill — การรีวิวโค้ดฉบับ Fable

> เป้าหมายของ review: **จับบั๊กก่อนถึง user + ยกระดับโค้ดเบส + กระจายความรู้ในทีม** — ไม่ใช่จับผิดหรืออวดภูมิ

---

## 1. ลำดับความสำคัญในการรีวิว (สูง → ต่ำ)

1. **Correctness** — โค้ดทำสิ่งที่ตั้งใจจริงไหม? edge case พังไหม? มี race condition ไหม?
2. **Security** — input จากภายนอกถูก validate ไหม? authz ครบไหม? secret รั่วไหม? (ใช้คำถามใน security.md ข้อ 6)
3. **Data safety** — migration ย้อนกลับได้ไหม? มีทาง data loss ไหม? backward compatible ไหม?
4. **Design** — ชิ้นนี้วางถูกที่ไหม? จะขวางการเปลี่ยนแปลงในอนาคตไหม? ซ้ำกับของที่มีอยู่ไหม?
5. **Tests** — มี test ครอบพฤติกรรมใหม่ไหม? test เข้มจริงหรือแค่พิธีกรรม?
6. **Readability** — คนที่ไม่ได้เขียนอ่านรู้เรื่องไหมในอีก 6 เดือน?
7. **Style** — เรื่อง format ควรเป็นหน้าที่ของ linter/formatter ไม่ใช่มนุษย์ ถ้ายังเถียงเรื่องนี้ในรีวิว = ไปตั้ง CI

> ถ้าเวลาน้อย: รีวิวข้อ 1–3 ให้แน่น ดีกว่าไล่ครบทุกข้อแบบผิว ๆ

---

## 2. วิธีอ่าน diff อย่างเป็นระบบ

1. **อ่าน description/ticket ก่อน** — ต้องรู้ว่าโค้ดนี้ "ควร" ทำอะไร ก่อนตัดสินว่ามัน "ทำจริง" ไหม
2. รอบแรก: **skim ทั้ง PR** ดูภาพรวม — ไฟล์ไหนคือใจกลาง ไฟล์ไหนคือ noise (rename, generated)
3. รอบสอง: อ่านไฟล์ใจกลางแบบละเอียด **เริ่มจาก interface/signature → data model → logic → tests**
4. อ่าน test ก่อนอ่าน implementation ก็ได้ — test บอกว่าผู้เขียนคิดว่าพฤติกรรมคืออะไร แล้วดูว่า implementation ตรงไหม
5. **มองหาสิ่งที่ "ไม่อยู่ใน diff"**: จุดเรียกใช้เดิมที่ควรถูกแก้แต่ไม่ถูกแก้, error path ที่หาย, test ที่ควรมี, doc ที่ stale, การลบ feature เก่าที่ไม่สมบูรณ์
6. ถ้า PR ใหญ่เกิน ~400 บรรทัด: ขอให้แตก — คุณภาพรีวิวตกฮวบตามขนาด

## 3. คำถามจับบั๊กประจำตัว (ยิงใส่ทุก diff)

```
□ ค่า null/empty/undefined เข้าฟังก์ชันนี้ได้ไหม แล้วเกิดอะไรขึ้น?
□ ถ้า exception เกิดกลางทาง state ค้างครึ่ง ๆ กลาง ๆ ไหม? (partial write, lock ไม่คืน)
□ เรียกซ้ำ 2 ครั้งพร้อมกัน / double-click พังไหม? idempotent ไหม?
□ Loop นี้ขอบถูกไหม? กับ n=0 และ n=1 ยังถูกไหม?
□ การเปลี่ยน signature/behavior นี้ พังผู้เรียกเดิมที่ไหนบ้าง? (grep ครบยัง)
□ Migration/config ใหม่: deploy แล้ว rollback ได้ไหม? รันบนข้อมูลจริงขนาด production ไหวไหม?
□ Query ใน loop ไหม (N+1)? มี index รับ query ใหม่ไหม?
□ Timezone/เงิน/float ถูกจัดการถูกประเภทไหม?
□ Resource (file, connection, lock) ถูกปิดทุก path รวมถึง error path ไหม?
□ Log/error message เผยข้อมูลอ่อนไหวไหม?
```

## 4. ศิลปะการให้ comment

- **Comment ที่โค้ด ไม่ใช่ที่คน**: "ตรงนี้จะ throw เมื่อ list ว่าง" ไม่ใช่ "คุณลืมเช็คอีกแล้ว"
- ระบุระดับความสำคัญให้ชัด เพื่อผู้เขียนจัดลำดับได้:
  - `[blocking]` — บั๊ก/ช่องโหว่ ต้องแก้ก่อน merge
  - `[suggestion]` — ดีขึ้นถ้าแก้ แต่ไม่ขวาง merge
  - `[nit]` — จุกจิก เอาไปคิดเฉย ๆ
  - `[question]` — ถามเพื่อเข้าใจ ไม่ได้บอกว่าผิด
- **อธิบาย why + เสนอทางออก** ไม่ใช่แค่ชี้: "แบบนี้พังเมื่อ X เพราะ Y — ลองใช้ Z ไหม"
- ชมของดีด้วย — เจอ solution สวย ๆ ก็บอก ทำให้วัฒนธรรมรีวิวไม่เป็นแดนลงโทษ
- ถ้าถกกันเกิน 3 รอบใน comment → ยกหูคุย/call แล้วสรุปผลกลับมาเขียนไว้
- อย่า approve สิ่งที่อ่านไม่เข้าใจ — "ไม่เข้าใจ" คือ signal ว่าโค้ดหรือ doc ต้องชัดกว่านี้

---

## 5. ฝั่งผู้ส่ง PR (ทำให้ตัวเองรีวิวง่าย = ได้รีวิวดี)

- **รีวิวตัวเองก่อนส่งเสมอ** — เปิด diff ไล่อ่านเหมือนเป็นคนอื่น จะเจอของหลุด 80% ก่อนใครเห็น
- PR เล็ก ธีมเดียว: refactor แยกจาก feature แยกจาก bugfix
- Description ตอบ 3 ข้อ: **ทำอะไร / ทำไม / ทดสอบยังไงแล้ว** + screenshot ถ้าเป็น UI
- ชี้จุดที่อยากให้ช่วยดูเป็นพิเศษ: "ไม่แน่ใจเรื่อง lock ตรง service.py:88"
- ตอบทุก comment (แม้แค่ "แก้แล้ว" / "ไม่แก้เพราะ...") — อย่าเงียบหาย, อย่าถือเป็นเรื่องส่วนตัว: เขารีวิวโค้ด ไม่ได้รีวิวคุณ

---

## 6. Checklist สรุปก่อนกด Approve

```
□ เข้าใจว่าโค้ดนี้ทำอะไรและทำไม (ถ้าไม่เข้าใจ ยังไม่ approve)
□ ไล่คำถามข้อ 3 ครบแล้ว ไม่มี blocking ค้าง
□ Test ใหม่มีจริง เข้มจริง และ CI เขียว
□ ไม่มี secret / debug code / TODO ที่ควรทำตอนนี้
□ ผลกระทบวงกว้าง (API contract, DB schema, dependency ใหม่) ถูกพิจารณาแล้ว
□ ถ้า merge แล้วพังตอนตี 3 — rollback ได้ และรู้ว่าต้อง rollback อะไร
```
